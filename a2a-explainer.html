Concepto base

¿Qué es el protocolo A2A?

A2A (Agent-to-Agent) es un protocolo abierto para que agentes de IA de distintos fabricantes y frameworks hablen entre sí de forma estándar. Igual que HTTP permite que cualquier navegador acceda a cualquier web, A2A permite que cualquier agente cliente llame a cualquier agente servidor.

Sin A2A

Cada sistema tiene su propio formato de API. Un agente de Microsoft no puede llamar a uno de Google sin código de integración ad-hoc.

Con A2A

Cualquier agente que implemente el protocolo puede comunicarse con cualquier otro, independientemente del framework o empresa.

En la práctica

El agente servidor expone un endpoint HTTP. El cliente lo descubre leyendo su "tarjeta de presentación" y luego le manda tareas via JSON-RPC.

Analogía sencilla A2A es como contratar a un especialista externo: el cliente publica una tarea, el especialista la ejecuta y devuelve el resultado — sin que el cliente sepa cómo trabaja por dentro.

¿Por qué importa?

  • Puedes construir un agente "orquestador" que delegue subtareas a agentes especializados.
  • Puedes reemplazar un agente por otro sin cambiar el cliente.
  • Funciona con streaming (respuestas en tiempo real) y tareas largas en background.
  • El estándar lo mantiene la comunidad — no está atado a ningún proveedor.
Estructura del sistema

Arquitectura cliente-servidor

Una comunicación A2A siempre tiene dos partes:

Cliente
envía tarea
──────▶
A2A
JSON-RPC / HTTP
──────▶
Servidor / Agente
ejecuta y responde

Servidor (Agente)

Es el agente que tiene una habilidad. Expone un endpoint HTTP, publica su AgentCard en /.well-known/agent.json, y recibe tareas via JSON-RPC POST en /.

Cliente

Es quien quiere usar la habilidad. Primero lee la AgentCard del servidor para descubrir sus capacidades, luego envía mensajes usando el método message/send o message/stream.

¿Qué tecnologías usa?

  • JSON-RPC 2.0 — formato de las llamadas (petición + respuesta con ID único).
  • HTTP/S — transporte. El servidor es un endpoint HTTP normal.
  • Server-Sent Events (SSE) — para streaming: el servidor envía chunks del resultado a medida que los produce.
  • Starlette + uvicorn — en Python, el framework ASGI que maneja las conexiones HTTP.

  ──────                            ─────────────────

  2. POST / (JSON-RPC)            ──▶  { method: "message/send",
     message: "42 + 58"                 params: { message: { parts: [{ text: "42 + 58" }] } } }

  3. ◀── respuesta                     { result: { ... texto: "42 + 58 = 100" } }

  o bien:

  3b. ◀── SSE stream                   data: {"artifact": {"parts": [{"text": "4"}]}}

Descubrimiento

AgentCard — la tarjeta de presentación

Antes de enviar una tarea, el cliente necesita saber qué puede hacer el agente y cómo hablarle. Eso está en la AgentCard, un JSON que el servidor publica en /.well-known/agent.json.

Equivalente web Es como el archivo robots.txt o sitemap.xml — una URL conocida donde cualquier cliente puede ir a descubrir las capacidades del agente.

Ejemplo de AgentCard del demo

Esto es lo que devuelve GET http://localhost:5001/.well-known/agent.json:

{
  "name": "CalculatorAgent",
  "description": "Evaluates arithmetic expressions: + - * / ** % ( )",
  "version": "1.0.0",
  "capabilities": {
    "streaming": true,       ← soporta SSE
    "pushNotifications": false
  },
  "skills": [{
    "id": "calculate",
    "name": "Calculate",
    "description": "Evaluate an arithmetic expression and return the result.",
    "examples": ["42 + 58", "What is 3.14 * 2?"]
  }],
  "supportedInterfaces": [{
    "url": "http://localhost:5001/",
    "protocolBinding": "JSONRPC"

}

Ciclo de vida

Flujo completo de una tarea

Cuando el cliente envía un mensaje, el servidor lo convierte en una Task (tarea). Las tareas tienen estados y el servidor va actualizando el cliente.

1

Cliente envía mensaje

El cliente llama a agent.run("42 + 58"). Por debajo, el SDK serializa el texto en un objeto Message y lo manda via JSON-RPC al endpoint del servidor.

2

Servidor crea una Task (WORKING)

El DefaultRequestHandler recibe el mensaje, genera un task_id único, y llama al AgentExecutor.execute(). El ejecutor primero encola un evento Task con estado WORKING para señalizar que empezó.

3

Servidor produce artefactos

El ejecutor encola eventos TaskArtifactUpdateEvent con fragmentos del resultado. En el demo, envía la respuesta carácter a carácter. En un agente LLM, enviaría tokens.

4

Servidor cierra con COMPLETED

Al terminar, el ejecutor encola TaskStatusUpdateEvent con estado COMPLETED. El framework cierra el stream SSE (si lo hay) y la tarea queda archivada en el InMemoryTaskStore.

5

Cliente recibe la respuesta

Según el patrón elegido (sync, stream, poll), el cliente obtiene el texto final ensamblado a partir de todos los artefactos.

Estados de una Task

  ┌──────────────┐     ┌─────────────┐     ┌───────────────┐
  │   SUBMITTED  │────▶│   WORKING   │────▶│   COMPLETED   │
  └──────────────┘     └─────────────┘     └───────────────┘
                              │
                              ├──────────────────────────────▶ CANCELED
                              │
                              └──────────────────────────────▶ FAILED
Comunicación

Los 3 patrones de comunicación

A2A soporta tres formas distintas de recibir la respuesta. Los tres usan el mismo servidor — la diferencia está en cómo el cliente hace la llamada.

Patrón 1 Non-streaming (síncrono)

El cliente envía la tarea y espera bloqueado hasta que el servidor termina. Aunque el servidor internamente use streaming y vaya enviando artefactos, el SDK del cliente los ensambla y devuelve el resultado final completo de golpe.

Cuándo usarlo: tareas cortas donde no necesitas ver el progreso — cálculos, consultas, respuestas rápidas. 

# agent.run() sin stream=True → espera y devuelve resultado completo
response = await agent.run("42 + 58")
print(response.text)   # "42 + 58 = 100"
  Cliente           Servidor
  ───────           ────────
  run("42 + 58") ──▶  [WORKING] evalúa...
                   ◀──  artefacto: "4"
                   ◀──  artefacto: "2"
                   ◀──  artefacto: " + 58 = 100"
                   ◀──  [COMPLETED]
  response.text = "42 + 58 = 100"  (ensamblado por el SDK)
Patrón 2 Streaming (SSE)

El cliente recibe cada fragmento del resultado a medida que el servidor lo produce. Usa Server-Sent Events (SSE): la conexión HTTP se mantiene abierta y el servidor va empujando datos. Ideal para mostrar texto "escribiéndose" en tiempo real.

Cuándo usarlo: respuestas largas de LLM, interfaces tipo chat donde quieres que el usuario vea el texto aparecer progresivamente. 

# stream=True → itera sobre actualizaciones en tiempo real
stream = agent.run("999 * 999", stream=True)
async for update in stream:
    for content in update.contents:
        if content.text:
            print(content.text, end="", flush=True)
# imprime: 9  9  9     *     9  9  9     =     9  9  8  0  0  1
# (carácter a carácter conforme llegan)

final = await stream.get_final_response()
print(final.text)   # "999 * 999 = 998001"
  Cliente           Servidor  (conexión SSE abierta)
  ───────           ──────────────────────────────
  run(stream=True) ──▶  [WORKING]
                   ◀──  SSE: "9"   (char 0)
  print "9"
                   ◀──  SSE: "9"   (char 1)
  print "9"
                   ◀──  SSE: "9"   (char 2)
  print "9"
                   ◀──  SSE: " * 999 = 998001"
  print " * 999 = 998001"
                   ◀──  SSE: [COMPLETED]
  conexión cerrada
Patrón 3 Background + Polling

El cliente dispara la tarea y el servidor responde inmediatamente con un continuation token (token de seguimiento). El cliente sigue haciendo otras cosas y periódicamente pregunta "¿ya terminaste?" usando ese token.

Cuándo usarlo: tareas muy largas (minutos, horas), sistemas donde el cliente puede desconectarse y reconectarse más tarde. 

# background=True → el servidor responde inmediatamente con token
response = await agent.run("2 ** 32", background=True)

if response.continuation_token is None:
    print("Completado al instante:", response.text)
else:
    token = response.continuation_token
    while token is not None:
        print("Tarea en progreso, esperando...")
        await asyncio.sleep(0.5)
        response = await agent.poll_task(token)
        token = response.continuation_token
    print("Resultado:", response.text)   # "2 ** 32 = 4294967296"
  Cliente                              Servidor
  ───────                              ────────
  run(background=True) ─────────────▶  inicia tarea, devuelve token
  ◀─ { continuation_token: "task-xyz" }

  (el cliente hace otras cosas)

  poll_task("task-xyz") ────────────▶  ¿terminaste? → No
  ◀─ { continuation_token: "task-xyz" }

  poll_task("task-xyz") ────────────▶  ¿terminaste? → Sí
  ◀─ { text: "2 ** 32 = 4294967296", continuation_token: None }
El código del demo

Cómo está construido el demo

El demo tiene dos archivos principales en demo/.

server.py — el agente servidor

La pieza más importante es el executor: la clase que implementa la lógica del agente.

class CalculatorExecutor(AgentExecutor):
    # AgentExecutor es una clase abstracta del a2a-sdk.
    # En los ejemplos de Microsoft, A2AExecutor() hace lo mismo pero
    # envolviendo un agente LLM (Azure/OpenAI). Aquí lo implementamos
    # directamente con Python puro.

    async def execute(self, context: RequestContext, event_queue: EventQueue) -> None:

        # 1. Obtenemos el texto que escribió el usuario
        user_text = context.get_user_input()   # "42 + 58"

        # 2. Señalizamos que empezamos (Task en estado WORKING)
        await event_queue.enqueue_event(
            Task(id=task_id, status=TaskStatus(state=TaskState.TASK_STATE_WORKING))
        )

        # 3. Evaluamos la expresión matemática (safe_eval, sin eval() directo)
        result_str = safe_eval(extract_expression(user_text))

        # 4. Enviamos el resultado carácter a carácter (demuestra streaming)
        for i, char in enumerate(result_str):
            await event_queue.enqueue_event(
                TaskArtifactUpdateEvent(artifact=Artifact(parts=[Part(text=char)]),
                                         append=(i > 0), last_chunk=(i == len(result_str)-1))
            )
            await asyncio.sleep(0.04)   # simula trabajo lento

        # 5. Marcamos la tarea como completada
        await event_queue.enqueue_event(
            TaskStatusUpdateEvent(status=TaskStatus(state=TaskState.TASK_STATE_COMPLETED))
        )
¿Por qué char a char? Para que el streaming sea visible en el demo. Un agente LLM real envía tokens (palabras o fragmentos de palabras) a medida que el modelo los genera. El principio es exactamente el mismo.

Evaluador matemático seguro

En vez de usar eval("42 + 58") (peligroso — permite ejecutar código arbitrario), el demo usa el AST (árbol sintáctico) de Python: parsea la expresión, recorre el árbol y solo permite nodos de tipo número y operadores aritméticos.

# En vez de: eval("42 + 58")  ← PELIGROSO
# Usamos: ast.parse() → recorrer árbol → solo números y operadores

def safe_eval(expression: str) -> float:
    tree = ast.parse(expression, mode="eval")
    return _eval_node(tree.body)

def _eval_node(node):
    if isinstance(node, ast.Constant):  # número → lo devolvemos
        return float(node.value)
    if isinstance(node, ast.BinOp):   # operación binaria: a + b
        op_fn = _SAFE_OPS[type(node.op)]  # solo +, -, *, /, **, %
        return op_fn(_eval_node(node.left), _eval_node(node.right))
    raise ValueError(f"No permitido: {type(node).__name__}")

Referencia

Glosario A2A

AgentCard
JSON de descubrimiento del agente. Se publica en /.well-known/agent.json. Contiene nombre, descripción, skills, URL del endpoint y capacidades (streaming, push notifications).
# Así se construye en Python (server.py)
AgentCard(
    name="CalculatorAgent",
    description="Evaluates arithmetic expressions",
    version="1.0.0",
    capabilities=AgentCapabilities(streaming=True),
    supported_interfaces=[AgentInterface(url="http://localhost:5001/", protocol_binding="JSONRPC")],
    skills=[AgentSkill(id="calculate", name="Calculate", ...)]
)

# Así se ve en el navegador → GET http://localhost:5001/.well-known/agent.json
{ "name": "CalculatorAgent", "version": "1.0.0",
  "capabilities": { "streaming": true },
  "skills": [{ "id": "calculate", "name": "Calculate" }] }
AgentSkill
Una habilidad específica que expone el agente. Tiene ID, nombre, descripción y ejemplos de uso. Un agente puede tener múltiples skills.
# Definición de una skill en el servidor
AgentSkill(
    id="calculate",               # ID único — lo usa el cliente para seleccionar
    name="Calculate",
    description="Evaluate an arithmetic expression",
    tags=["math", "calculator"],
    examples=["42 + 58", "What is 3.14 * 2?"]
)

# El cliente puede ver las skills disponibles tras resolver la AgentCard
for skill in agent_card.skills:
    print(skill.name, "→", skill.description)
# Calculate → Evaluate an arithmetic expression
Task
Unidad de trabajo en A2A. Tiene un ID único y pasa por estados: WORKING → COMPLETED / CANCELED / FAILED. Puede llevar artefactos (la respuesta).
# El executor encola una Task al inicio para "registrar" el trabajo
await event_queue.enqueue_event(
    Task(
        id="task-abc-123",        # ID único generado por el framework
        context_id="ctx-xyz",    # agrupa tareas relacionadas
        status=TaskStatus(state=TaskState.TASK_STATE_WORKING)
    )
)
# Desde ese momento el cliente sabe que hay una tarea viva con ese ID
# y puede hacer polling con ese ID si usa background=True
AgentExecutor
Interfaz abstracta que todo servidor A2A debe implementar. Define el método execute(context, event_queue) — ahí va la lógica del agente (LLM, reglas, lo que sea).
# La interfaz que debes implementar (del a2a-sdk)
class AgentExecutor(ABC):
    async def execute(self, context: RequestContext, event_queue: EventQueue) -> None: ...
    async def cancel(self, context: RequestContext, event_queue: EventQueue) -> None: ...

# Tu implementación (demo/server.py)
class CalculatorExecutor(AgentExecutor):
    async def execute(self, context, event_queue):
        text = context.get_user_input()     # "42 + 58"
        result = safe_eval(text)             # 100.0
        await event_queue.enqueue_event(...)  # envía resultado

# Comparación: en los ejemplos de Microsoft usan A2AExecutor que envuelve un LLM
executor = A2AExecutor(Agent(client=FoundryChatClient(...), ...))
EventQueue
Cola de eventos que el executor usa para enviar resultados al cliente. Se encolan objetos Task, TaskArtifactUpdateEvent y TaskStatusUpdateEvent.
# El executor recibe event_queue como parámetro — solo puede escribir en ella
async def execute(self, context, event_queue):

    # 1. Señaliza inicio
    await event_queue.enqueue_event(Task(..., status=WORKING))

    # 2. Envía fragmentos del resultado
    await event_queue.enqueue_event(TaskArtifactUpdateEvent(...))
    await event_queue.enqueue_event(TaskArtifactUpdateEvent(...))

    # 3. Cierra la tarea
    await event_queue.enqueue_event(TaskStatusUpdateEvent(..., status=COMPLETED))

# El framework lee la cola y reenvía cada evento al cliente via SSE o JSON-RPC
TaskArtifactUpdateEvent
Evento que transporta un fragmento del resultado (texto, imagen, etc.). Tiene append=True para chunks adicionales y last_chunk=True en el último.
# Primer chunk (append=False): crea el artefacto
TaskArtifactUpdateEvent(
    task_id="task-123", context_id="ctx-xyz",
    artifact=Artifact(artifact_id="art-1", parts=[Part(text="4")]),
    append=False, last_chunk=False
)
# Chunks intermedios (append=True): añade al artefacto existente
TaskArtifactUpdateEvent(..., artifact=Artifact(parts=[Part(text="2")]),
    append=True, last_chunk=False)
TaskArtifactUpdateEvent(..., artifact=Artifact(parts=[Part(text=" + 58 = 100")]),
    append=True, last_chunk=True)  # ← último chunk

# El cliente ensambla: "4" + "2" + " + 58 = 100" = "42 + 58 = 100"
TaskStatusUpdateEvent
Evento de cambio de estado de la tarea. El más importante es el final con TASK_STATE_COMPLETED que cierra la tarea.
# Al terminar el trabajo, el executor DEBE enviar este evento
# De lo contrario el cliente espera indefinidamente
await event_queue.enqueue_event(
    TaskStatusUpdateEvent(
        task_id="task-123",
        context_id="ctx-xyz",
        status=TaskStatus(state=TaskState.TASK_STATE_COMPLETED)
    )
)
# Otros estados posibles:
#   TASK_STATE_WORKING   → tarea en curso (al inicio)
#   TASK_STATE_COMPLETED → terminó con éxito
#   TASK_STATE_CANCELED  → cancelada por el cliente
#   TASK_STATE_FAILED    → error (el framework lo pone si execute() lanza excepción)
continuation_token
Token opaco que identifica una tarea en curso. Devuelto cuando se usa background=True. Se pasa a poll_task() para consultar el progreso.
# Cuando background=True y la tarea sigue corriendo, response tiene token
response = await agent.run("2 ** 32", background=True)

print(response.continuation_token)
# { "task_id": "task-abc-123", "context_id": "ctx-xyz" }
# Es un dict con los IDs necesarios para retomar la tarea

# Si la tarea terminó antes de que preguntaras, el token es None
if response.continuation_token is None:
    print("Ya terminó:", response.text)   # respuesta inmediata
else:
    response = await agent.poll_task(response.continuation_token)
A2ACardResolver
Clase del SDK cliente que descarga y parsea la AgentCard de un servidor A2A dado su URL base.
# Siempre el primer paso antes de crear un A2AAgent
async with httpx.AsyncClient(timeout=60.0) as http_client:
    resolver = A2ACardResolver(
        httpx_client=http_client,
        base_url="http://localhost:5001/"
    )
    agent_card = await resolver.get_agent_card()
    # ↑ hace GET http://localhost:5001/.well-known/agent.json
    # y devuelve un objeto AgentCard ya parseado

print(agent_card.name)          # "CalculatorAgent"
print(agent_card.capabilities)  # AgentCapabilities(streaming=True)
A2AAgent
Clase del SDK cliente que encapsula toda la comunicación con un servidor A2A. Expone .run() con los tres patrones (sync, stream, background).
# Se crea con los datos de la AgentCard y la URL del servidor
async with A2AAgent(
    name=agent_card.name,
    description=agent_card.description,
    agent_card=agent_card,
    url="http://localhost:5001/",
) as agent:

    # Los 3 patrones — mismo objeto, distinto .run()
    r1 = await agent.run("42 + 58")                          # sync
    s  = agent.run("42 + 58", stream=True)                  # streaming
    r3 = await agent.run("42 + 58", background=True)        # polling

# A2AAgent también puede llamar a una skill específica como herramienta:
#   tool = agent.as_tool(name="Calculate", description="...")
# Eso permite que un LLM decida cuándo invocar el agente remoto.
DefaultRequestHandler
El "router" del servidor A2A. Recibe las llamadas JSON-RPC, gestiona el ciclo de vida de las tareas, llama al AgentExecutor, y envía los eventos de vuelta al cliente.
# Se construye con 3 piezas: tu executor, el store de tareas, y la AgentCard
request_handler = DefaultRequestHandler(
    agent_executor=CalculatorExecutor(),   # ← tu lógica
    task_store=InMemoryTaskStore(),          # ← donde guarda las tareas
    agent_card=AGENT_CARD,                    # ← la tarjeta del agente
)

# Internamente hace todo esto por ti:
#   1. Parsea el JSON-RPC que llega por HTTP
#   2. Crea una Task en el task_store
#   3. Llama a executor.execute(context, event_queue)
#   4. Lee los eventos de la cola y los reenvía al cliente
#   5. Actualiza el estado de la Task en el store
InMemoryTaskStore
Almacén de tareas en memoria RAM (se pierde al reiniciar el servidor). En producción se reemplazaría por un store persistente (Redis, base de datos).
# Uso básico — normalmente no lo tocas tú directamente
task_store = InMemoryTaskStore()
# El DefaultRequestHandler lo usa internamente para:
#   - Guardar tareas en curso
#   - Recuperarlas cuando el cliente hace poll por task_id
#   - Actualizarlas con cada TaskStatusUpdateEvent

# ¿Por qué importa el store?
# Si el cliente se desconecta durante una tarea larga y después
# vuelve con el continuation_token, el servidor busca la tarea
# en el store para saber dónde está y reanudar el stream.

# En producción, reemplazarías esto por un store Redis:
# task_store = RedisTaskStore(redis_url="redis://localhost:6379")
JSON-RPC 2.0
Protocolo de llamada a procedimientos remotos sobre JSON. Cada petición tiene un method (ej. "message/send"), params y un id para correlacionar respuestas.
# Así se ve una petición A2A real por HTTP (lo hace el SDK, no tú)
# POST http://localhost:5001/
{
  "jsonrpc": "2.0",
  "id": "req-001",           ← para correlacionar petición con respuesta
  "method": "message/send",   ← qué operación
  "params": {
    "message": {
      "role": "user",
      "parts": [{ "text": "42 + 58" }]
    }
  }
}

# Para streaming se usa "message/stream" en vez de "message/send"
# Para polling se usa "tasks/get" con el task_id
SSE (Server-Sent Events)
Tecnología HTTP estándar donde el servidor mantiene la conexión abierta y va enviando mensajes al cliente. Más simple que WebSockets para flujos unidireccionales (servidor → cliente).
# Así se ve el wire format SSE real (texto plano por HTTP)
# El servidor envía líneas "data: ..." separadas por \n\n

data: {"artifact":{"artifactId":"a1","parts":[{"text":"4"}]},"append":false}

data: {"artifact":{"artifactId":"a1","parts":[{"text":"2"}]},"append":true}

data: {"artifact":{"artifactId":"a1","parts":[{"text":" + 58 = 100"}]},"lastChunk":true}

data: {"status":{"state":"TASK_STATE_COMPLETED"}}

# El cliente recibe cada "data:" como un evento separado en tiempo real
# agent_framework.a2a.A2AAgent los parsea y te los entrega como objetos Python
A2AExecutor
Implementación de AgentExecutor que envuelve un Agent del framework de Microsoft. Traduce entre la API del framework y los eventos A2A. El demo no lo usa — implementa AgentExecutor directamente.
# Usado en los ejemplos de Microsoft (04-hosting_a2a/)
# Requiere Azure AI Foundry o OpenAI
agent = Agent(
    client=FoundryChatClient(project_endpoint="...", model="gpt-4o", credential=AzureCliCredential()),
    name="PolicyAgent",
    instructions="Siempre responde con la política de devoluciones...",
)
executor = A2AExecutor(agent, stream=True)  # ← envuelve el Agent en un AgentExecutor A2A

# El demo implementa AgentExecutor directamente
executor = CalculatorExecutor()  # ← mismo interface
